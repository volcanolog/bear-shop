Практическая работа №9: Refresh-токены и ротация сессий

Цель: Реализация отказоустойчивой системы авторизации с использованием механизмов обновления токенов (Refresh Token) и защиты от повторного использования (Rotation).

Что добавлено:

    Двухуровневая система токенов:

        Внедрено разделение на Access Token (короткоживущий, 15 мин) и Refresh Token (долгоживущий, 7 дней) для минимизации рисков при краже ключей.

        Реализованы разные секретные ключи для подписи каждого типа токена.

    Механизм ротации (Token Rotation):

        Разработан эндпоинт POST /api/auth/refresh, который при каждом обновлении аннулирует старый refresh-токен и выдает новую пару.

        Реализовано серверное хранилище активных refresh-токенов для возможности мгновенного отзыва всех сессий пользователя.

    Автоматизация на стороне фронтенда (Axios Interceptors):

        Настроен Request Interceptor для автоматической подстановки актуального токена в заголовки всех запросов.

        Реализован Response Interceptor, который незаметно для пользователя обновляет сессию при получении ошибки 401 и повторяет неудавшееся действие.

    Обработка истечения сессии:

        Добавлена логика принудительного логаута и очистки localStorage в случае невалидности или истечения срока действия refresh-токена.

    Тестирование жизненного цикла:

        Проведена проверка сценария «использования старого refresh-токена»: подтверждено, что система блокирует доступ при попытке повторного использования уже обновленного ключа.

Практическая работа №11: Управление доступом на основе ролей (RBAC)

Цель: Разграничение прав пользователей системы по ролям (гость / user / seller / admin).

Что добавлено:

    Роли и хранение:
        Поля `role` и `isBlocked` у пользователя.
        Первый зарегистрированный автоматически получает роль admin (учебный приём).
        Назначение ролей в дальнейшем — только через PUT /api/users/:id (доступно admin).

    Серверные middleware:
        authMiddleware — проверка валидности access-токена и того, что пользователь не заблокирован.
        roleMiddleware([ROLES...]) — фабрика middleware, ограничивающая доступ конкретными ролями.
        Роль кладётся в payload JWT (поле role) — сервер не ходит в БД ради проверки прав.

    Защита эндпоинтов согласно таблице задания:
        /api/auth/register, /api/auth/login, /api/auth/refresh — гость;
        /api/auth/me — любая авторизованная роль;
        /api/users, /api/users/:id (GET/PUT/DELETE) — только admin;
        /api/products GET — любая авторизованная роль;
        /api/products POST, PUT — seller и admin;
        /api/products DELETE — только admin.

    Блокировка пользователей:
        DELETE /api/users/:id выполняет «мягкое удаление» — выставляет isBlocked=true.
        Заблокированный не может ни залогиниться, ни пройти authMiddleware с ранее выданным токеном.
        Защита от блокировки самого себя (на сервере и в UI).

    Ротация refresh-токенов:
        При каждом /api/auth/refresh старый токен инвалидируется, выдается новая пара.
        Серверное хранилище активных refresh-токенов (Set) позволяет «отозвать» сессию.

    Фронтенд:
        Кнопки «+ Создать», «Редактировать», «Удалить» отображаются только при наличии роли.
        Админу доступна страница «Пользователи» с таблицей: смена роли и блокировка.
        Запрос /api/auth/me используется как источник истины о текущей роли (а не данные с экрана логина).
